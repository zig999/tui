## 1. Purpose and Responsibilities

Renders a determinate progress bar as two nested `<div>`s: an outer track (`role="progressbar"`, fixed height, `bg-surface`) and an inner fill whose width is computed from `value`/`max` and set via an inline `style` (not a token, since it is a continuous data value). Optionally renders a header row with a `label` and/or the rounded percentage (`showValue`). Applies a `tone` color to the fill via a single-axis `cva()`.

**Out of scope for this component:** indeterminate/loading-spinner presentation (no `indeterminate` prop or animation for unknown progress), driving the `value` over time (fully controlled — no internal timer/interval), validating that `value`/`max` are sane inputs beyond clamping for display.

---

## 2. When to Use / When Not to Use

| Use when | Do not use when |
|----------|-----------------|
| Displaying determinate progress of a known quantity (e.g. upload percentage, multi-step completion) with a numeric `value` out of `max` | Progress is unknown/indeterminate (no meaningful `value`) → this component has no indeterminate mode <!-- TO CONFIRM: no separate indeterminate-progress component exists in this UI Kit --> or a loading placeholder is needed instead of a metered bar → use `Skeleton` (`skeleton.component.spec.md`) |

## 3. Props Contract

```ts
import type { ComponentProps } from "react";
```

| Prop | Type | Required | Default | Description |
|------|------|----------|---------|-------------|
| `value` | `number` | **yes** | — | Current value (`0..max`); clamped to `[0, max]` before rendering (see §4 transition parameters) |
| `max` | `number` | no | `100` | Maximum value used as the denominator for the fill percentage and `aria-valuemax` |

| `className` | `string` | no | `undefined` | Merged onto the root wrapper `<div>` via `cn()` (not onto the track or fill) |
| ...props | `ComponentProps<"div">` | no | — | + native `<div>` attributes, spread onto the root wrapper `<div>` |

## 4. Component States

> No internal state (no `useState`/`useReducer`). All states below are prop-driven presentation recomputed on every render from `value`/`max`.

| State | Trigger | Visual change | Interactivity |
|-------|---------|---------------|----------------|
| `idle` | initial render, `0 < value < max` | fill width set to `${percent * 100}%`; track shows partial fill in the `tone` color | none (not interactive; no keyboard/pointer handlers) |
| `empty` | `value <= 0` (clamped to `0`) | fill width is `0%` — fill `<div>` renders with zero width | none |
| `full` | `value >= max` (clamped to `max`) | fill width is `100%` — fill fully covers the track | none |
| `labeled header` | `label` and/or `showValue` truthy | header row renders above the track: `label` left-aligned (`text-xs text-muted-foreground`), percentage right-aligned (`text-xs text-foreground`, `ml-auto`) when `showValue` | none |

> **Transition parameters:** the fill width is a continuous formula, not a token — documented exactly as implemented.

| Parameter | Formula / Value | Unit | Applies to state |
|-----------|----------------|------|-----------------|
| `clamped` | `Math.min(Math.max(value, 0), max)` | number | all states |
| `percent` | `max > 0 ? clamped / max : 0` | ratio (0..1) | all states |
| `fill width` | `` `${percent * 100}%` `` (inline `style.width`, unrounded) | `%` | all states |
| `percentLabel` (display only) | `` `${Math.round(percent * 100)}%` `` | `%` | `labeled header` (only when `showValue`) |

<!-- TO CONFIRM: if max <= 0, percent is forced to 0 regardless of value (division-by-zero guard) — the fill renders at 0% and percentLabel shows "0%"; not exercised by any story. -->

## 6. Variants and Compositions

| Variant | Prop | Token classes | Usage context |
|---------|------|----------------|----------------|
| `default` | `tone="default"` (default) | `bg-primary` | <!-- TO CONFIRM: neutral/in-progress state --> |
| `success` | `tone="success"` | `bg-success` | <!-- TO CONFIRM: completed/positive outcome (see Success story: value=100, tone="success") --> |
| `warning` | `tone="warning"` | `bg-warning` | <!-- TO CONFIRM: attention-needed state (see Warning story: value=70, tone="warning") --> |
| `destructive` | `tone="destructive"` | `bg-destructive` | <!-- TO CONFIRM: failure/error state (see Destructive story: value=25, tone="destructive") --> |

**Fill base classes (all tones):** `h-full transition-[width] duration-150 motion-reduce:transition-none`

**Track classes (fixed, not a variant):** `h-2 w-full bg-surface`

**Root wrapper classes (fixed, not a variant):** `flex flex-col gap-1`

**Default variants:** `tone: "default"`

**Fill width mechanism:** the fill `<div>`'s width is set exclusively via an inline `style={{ width: `${percent * 100}%` }}` — it is the one dimension in this component intentionally not expressed as a semantic token, since it derives from a continuous runtime `value`/`max` ratio rather than a discrete design scale.

## 7. Do / Don't

| Do | Don't |
|----|-------|
| Pass `label` when the bar's meaning isn't obvious from context (it also becomes the track's `aria-label`) | Omit both `label` and an external accessible name when the bar's purpose isn't clear from surrounding context — the track has no fallback text if `label` is absent |
| Use `showValue` together with `label` for a header row showing both context and percentage (see `WithLabelAndValue` story) | Pass a `value` outside `[0, max]` expecting the bar to visually overflow — it is silently clamped, not rejected |

## 8. BDD Scenarios

### Default render

```
Given value=40 and no other props (max defaults to 100, tone defaults to "default")
When it mounts
Then the track renders role="progressbar" aria-valuenow={40} aria-valuemin={0} aria-valuemax={100}, and the fill div has style width: "40%" with bg-primary
```

### Out-of-range value (key state)

```
Given value is passed greater than max (e.g. value=150, max=100)
When it mounts
Then clamped is computed as Math.min(Math.max(150, 0), 100) = 100, the fill renders at width: "100%", and aria-valuenow is 100
```

### Keyboard navigation

```
Given the Progress component is rendered on the page
When the user tabs through the page with the keyboard
Then focus never lands on the progressbar — it is a non-interactive div with role="progressbar" and no tabIndex, so it is not part of the tab order
```

## 9. Accessibility Contract

| Requirement | Implementation |
|-------------|-----------------|
| Label | `aria-label={label}` on the track element; if `label` is `undefined`, no `aria-label` attribute is rendered (React omits `undefined` attributes) — the bar then has no accessible name |
| Keyboard | None — the track is a plain `<div role="progressbar">` with no `tabIndex`, not part of the tab order, no keyboard handlers |
| Focus management | Not applicable — the component never receives or manages focus |
| ARIA states | `role="progressbar"`, `aria-valuenow={clamped}`, `aria-valuemin={0}`, `aria-valuemax={max}` always set on the track |
