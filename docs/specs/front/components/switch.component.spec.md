## 1. Purpose and Responsibilities

A binary on/off toggle built on a native `<button type="button" role="switch">` (not a checkbox), styled as a fixed-width terminal-style pill that reads literally `[ ON ]` / `[ OFF ]`, optionally paired with a trailing text `label`. Fully controlled ‚Äî clicking calls `onChange(!checked)` but the component never updates its own visual state; the consumer must feed the new `checked` value back in.

**Out of scope for this component:** any internal toggled state (there is none ‚Äî see ¬ß4); grouping/label association beyond the plain sibling `<span>` inside the wrapping `<label>`.

---

## 2. When to Use / When Not to Use

| Use when | Do not use when |
|----------|-----------------|
| An immediate binary on/off setting that takes effect right away (no form submission step implied) | The control is part of a native HTML form that must submit a checkbox value ‚Üí use `checkbox` instead |
| A single independent boolean setting | Choosing one of several mutually-exclusive options ‚Üí use `radio-group` or `select` instead |

## 4. Component States

> No `useState`/`useReducer` ‚Äî the component holds no internal state at all. The on/off visual is a direct, one-to-one mapping of the `checked` prop (see ¬ß3); clicking does not change anything locally.

| State | Trigger | Visual change | Interactivity |
|-------|---------|---------------|---------------|
| `idle` | initial render | Renders per `checked`/`disabled` props only ‚Äî no internal React state exists in this component | full |

## 5. Events Emitted

| Event | Payload type | When emitted | Consumer action |
|-------|-------------|--------------|-----------------|
| `onChange` | `(checked: boolean) => void` | Native `click` on the `<button>` (mouse click, or Space/Enter ‚Äî native button activation) | Update the controlled `checked` prop to the emitted value; the switch will not visually flip on its own |

## 6. Variants and Compositions

> No `cva()` call exists in `switch.tsx` ‚Äî there is no `variant`/`size` prop. Token classes are applied per `checked`/`aria-invalid` state:

| Variant | Prop | Usage context |
|---------|------|---------------|
| `on` | `checked === true` | `border-primary bg-primary text-primary-foreground`, text content `[ ON ]` |
| `off` | `checked === false` | `border-border bg-background text-muted-foreground hover:border-border-strong`, text content `[ OFF ]` |
| `invalid` | `aria-invalid` passthrough attribute | `aria-invalid:border-destructive` (Tailwind's built-in `aria-*` variant, reads the native `aria-invalid` attribute) |
| `disabled` | `disabled === true` | Wrapping `<label>`: `cursor-not-allowed opacity-50`; button gets native `disabled` (no explicit disabled className branch beyond the label) |

## 7. Do / Don't

| Do | Don't |
|----|-------|
| Treat this as a fully controlled component ‚Äî update `checked` from `onChange` yourself | Don't assume clicking toggles the visual automatically; without state update from the consumer the button stays in its current `checked` value |
| Use `label` for the adjacent text so it stays inside the same `<label>` wrapper (button is labelable, so the whole row is clickable) | Don't pass `children` ‚Äî the prop type omits it (`children` excluded from the native passthrough) |

## 8. BDD Scenarios

### Default render

```
Given the component receives checked: false
When it mounts
Then it renders role="switch" aria-checked="false" with text "[ OFF ]" and border-border styling
```

### Disabled state

```
Given disabled: true
When it mounts
Then the wrapping label shows cursor-not-allowed opacity-50 and the button is natively disabled (unclickable, unfocusable)
```

### Keyboard navigation

```
Given the switch button is focused (native <button>, no custom key handling in this component)
When the user presses Space or Enter
Then the native browser fires a click, calling onChange with the inverted checked value
```

## 9. Accessibility Contract

| Requirement | Implementation |
|-------------|---------------|
| Label | No `aria-label`; accessible name comes from the optional `label` text sibling inside the same `<label>` element (button is a labelable element, so implicit label association applies when `label` is set) ‚Äî if `label` is omitted, the button has no accessible name |
| Keyboard | No custom key handling; relies entirely on native `<button>` semantics (Space/Enter activate `onClick`, Tab moves focus) |
| Focus management | `focus-visible:ring-2 focus-visible:ring-ring` on the button itself |
| ARIA states | `role="switch"` (overrides the native `button` role) with `aria-checked={checked}`; `aria-invalid` accepted via passthrough and styled via `aria-invalid:border-destructive` |
