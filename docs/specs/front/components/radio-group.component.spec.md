## 1. Purpose and Responsibilities

Renders a single-selection radio control set as a compound component: `RadioGroup` (the container, providing shared selection state via React Context) and `RadioGroupItem` (one selectable option, rendered as a visually hidden native `<input type="radio">` behind a styled indicator `<span>`, wrapped in a `<label>`).

`RadioGroup` sets `role="radiogroup"` on its container `div` and supplies `name`, `value`, `onValueChange`, and `disabled` to every descendant `RadioGroupItem` through `RadioGroupContext`. `RadioGroupItem` reads that context, computes whether it is the selected item (`groupValue === value`) and whether it is disabled (`groupDisabled || itemDisabled`), and renders the native input plus a styled indicator.

**Out of scope for this component:** grouping options from a data array (no `options[]` prop — options are composed as JSX children); any non-semantic/raw-hex color theming (a prior `colored` palette-cycling feature was dropped during migration, see §Changelog note below); form validation logic (only exposes native `aria-invalid` pass-through styling).

## 2. When to Use / When Not to Use

| Use when | Do not use when |
|----------|-----------------|
| The user must pick exactly one option from a small, fully-visible set (2–6 options) rendered inline in a form | The option set is large or needs a collapsed/searchable UI → use a `Select` component instead <!-- TO CONFIRM: no Select component found in this UI kit tier --> |
| Options are known at compose-time and can be authored as JSX children | Options come from a dynamic/remote list that must be data-driven without JSX composition → this component intentionally does not support an `options[]` prop (dropped by design, see §1) |

## 3. Props Contract

### `RadioGroup`

| Prop | Type | Required | Default | Description |
|------|------|----------|---------|-------------|
| `name` | `string` | no | `useId()`-generated string | Shared `name` attribute applied to every descendant `RadioGroupItem`'s native input. If omitted, an internally generated id (via `React.useId()`) is used so radios remain grouped natively. |
| `value` | `string` | yes | — | Currently selected value; compared against each `RadioGroupItem`'s `value` to compute `checked`. |
| `onValueChange` | `(value: string) => void` | yes | — | Called with the new value when the user selects a different item. |
| `disabled` | `boolean` | no | `undefined` (falsy) | When `true`, disables every descendant item unless overridden per-item (per-item `disabled` is OR'ed with this group-level flag, not overridden — see §4). |
| `className` | `string` | no | — | Merged via `cn()` onto the container `div`. |
| `ref` | `Ref<HTMLDivElement>` | no | — | Forwarded as a normal prop (React 19) to the container `div`. |
| ...props | `Omit<ComponentProps<"div">, "role">` | no | — | + native `div` attributes (passthrough). `role` is excluded from passthrough — the component always sets `role="radiogroup"` itself. |

### `RadioGroupItem`

| Prop | Type | Required | Default | Description |
|------|------|----------|---------|-------------|
| `value` | `string` | yes | — | This item's value. Compared against the group's `value` to compute selection; passed to `onValueChange` on selection. |
| `disabled` | `boolean` | no | `undefined` (falsy) | Per-item disabled flag. Combined with the group's `disabled` via logical OR — an item is disabled if either the group or the item itself sets `disabled`. |
| `children` | `ReactNode` | no | — | Rendered as the visible label text/content, inside the same `<label>` as the input. |
| `id` | `string` | no | `` `${name}-${value}` `` (derived) | `id` of the native input and `htmlFor` target of the wrapping `<label>`. If omitted, computed from the group's resolved `name` and this item's `value`. |
| `className` | `string` | no | — | Merged via `cn()` onto the wrapping `<label>` element (not the input, not the indicator span). |
| `ref` | `Ref<HTMLInputElement>` | no | — | Forwarded as a normal prop (React 19) to the native `<input>`. |
| ...props | `Omit<ComponentProps<"input">, "type" \| "name" \| "checked" \| "onChange" \| "value">` | no | — | + native `input` attributes (passthrough), excluding `type`, `name`, `checked`, `onChange`, `value` — those five are fully controlled internally and cannot be overridden by the consumer. |

**Context (internal, not a public prop):** `RadioGroupContextValue = { name: string; value: string; onValueChange: (value: string) => void; disabled?: boolean }`. Provided by `RadioGroup` via `RadioGroupContext` (a `React.Context` created at module scope). `RadioGroupItem` reads it via an internal `useRadioGroupContext()` hook that **throws** `Error("RadioGroupItem must be used within a RadioGroup")` if rendered outside a `RadioGroup`. This is a hard runtime contract — `RadioGroupItem` must always be a descendant of `RadioGroup`.

## 4. Component States

> No `useState`/`useReducer` in either `RadioGroup` or `RadioGroupItem`. All states below are prop-driven or context-derived, listed here because the component computes and applies them itself (not passed through verbatim).

| State | Trigger | Visual change | Interactivity |
|-------|---------|---------------|----------------|
| `idle` (unselected) | `groupValue !== item.value`, not disabled | Indicator span: `border-border`, no fill; hover → `group-hover:border-border-strong` | full |
| `selected` | `groupValue === item.value` | Indicator span: `border-primary bg-primary`, inner dot `<span className="size-1.5 bg-primary-foreground" />` rendered | full |
| `disabled` | `groupDisabled \|\| itemDisabled` is `true` | Wrapping `<label>`: `cursor-not-allowed opacity-50`; native `input` gets `disabled` attribute | none |
| `focus-visible` | native input receives keyboard focus | Indicator span gains `ring-2 ring-ring ring-offset-2 ring-offset-background` via `peer-focus-visible:` (input is `peer`) | full |
| `invalid` | consumer passes `aria-invalid` on `RadioGroupItem` (native attribute passthrough) | Indicator span: `border-destructive` via `peer-aria-invalid:border-destructive` | full |

## 5. Events Emitted

| Event | Payload type | When emitted | Consumer action |
|-------|-------------|--------------|------------------|
| `onValueChange` | `(value: string) => void` | Native `<input>` `onChange` fires (user clicks the item or selects it via keyboard) — called with the clicked item's `value` | Consumer updates the value it controls and passes it back down as `RadioGroup`'s `value` prop (fully controlled component — there is no internal selection state) |

## 7. Do / Don't

| Do | Don't |
|----|-------|
| Always render `RadioGroupItem` as a descendant of `RadioGroup` | Render `RadioGroupItem` outside a `RadioGroup` — it throws at render time (`useRadioGroupContext`) |
| Control `value` and `onValueChange` from the parent (fully controlled) | Expect the component to hold its own selection state — it has none; omitting `onValueChange` handling leaves the UI unable to update |
| Set `disabled` on `RadioGroup` to disable the whole set | Set `disabled` only on the group expecting a single item to remain enabled — group `disabled` ORs with (does not override) item `disabled`, but does not let an item opt back in |
| Pass `aria-invalid` on individual `RadioGroupItem` for validation styling | Expect a group-level invalid state — validation styling is per-item only, driven by native `aria-invalid` passthrough |

## 8. BDD Scenarios

### Default render

```
Given a RadioGroup with value="free" and three RadioGroupItem children (free, pro, enterprise)
When it mounts
Then the "free" item's input is checked, the indicator span shows border-primary bg-primary with the inner dot, and the other items render unselected
```

### Disabled item

```
Given a RadioGroupItem with disabled={true} (e.g. "enterprise")
When it renders
Then its wrapping label has cursor-not-allowed and opacity-50, and its native input has the disabled attribute, blocking pointer and keyboard interaction
```

### Keyboard navigation and selection

```
Given the RadioGroup is rendered and the user tabs to a radio input
When the user presses Tab to focus an item and clicks (or presses Space/Arrow key per native radio input semantics) to select "pro"
Then onValueChange is called with "pro", and once the parent updates the controlled value prop back to "pro" the "pro" input becomes checked
```

### Rendered outside a RadioGroup

```
Given a RadioGroupItem is rendered without an ancestor RadioGroup
When it mounts
Then useRadioGroupContext throws Error("RadioGroupItem must be used within a RadioGroup")
```

## 9. Accessibility Contract

| Requirement | Implementation |
|-------------|----------------|
| Label | Each item's visible children are rendered inside the same `<label htmlFor={inputId}>` that wraps the native input — no separate `aria-label` is set; the label text is the accessible name |
| Keyboard | Native `<input type="radio">` semantics — Tab moves focus into/out of the radio group's native tab stop, arrow keys move selection between radios of the same `name` (browser-native behavior, not custom-coded) |
| Focus management | No custom focus trapping/redirection. Focus ring is applied via `peer-focus-visible:ring-2 peer-focus-visible:ring-ring peer-focus-visible:ring-offset-2 peer-focus-visible:ring-offset-background` on the indicator span (input itself is visually hidden via `peer sr-only`) |
| ARIA states | `role="radiogroup"` on the `RadioGroup` container; native `checked`/`disabled` attributes on each input carry their implicit ARIA state; `aria-hidden="true"` on the decorative indicator span; `aria-invalid` supported via native passthrough on the input (styled via `peer-aria-invalid:border-destructive`) |
