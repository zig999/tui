## 1. Purpose and Responsibilities

A single `Card` component with slot props (`header` / `headerTitle` / `footer`
/ `children`) and CVA-driven `tone`/`padding` variants. It optionally becomes a
fully interactive, keyboard-activatable element when an `onClick` prop is
supplied.

<!-- MIGRATION NOTE (from source): the original (2ndbrain) `card.tsx` exports
     a single Card with tone/header/headerTitle/footer/padding props — there is
     NO CardHeader/CardTitle/CardContent/CardFooter subcomponent family. This is
     migrated as-is; no shadcn-style subcomponents were invented. -->

**Out of scope for this component:** compound subcomponents (`CardHeader`,
`CardTitle`, `CardContent`, `CardFooter` do not exist in this source); nested
interactive-element guarding (no protection against putting a button/link
inside an `onClick` card — see §7).

## 2. When to Use / When Not to Use

| Use when | Do not use when |
|----------|-----------------|
| Grouping related content in a bordered surface, optionally with a header/footer slot | Content needs the shadcn-style `CardHeader`/`CardTitle`/`CardContent`/`CardFooter` compound API → not available here, use the slot props (`header`, `headerTitle`, `footer`) instead |
| The whole card should act as one clickable/keyboard-activatable target (`onClick` provided) | The card contains its own nested interactive elements (buttons/links) — combining with `onClick` produces overlapping interactive targets (see §7) |

## 3. Props Contract

| Prop | Type | Required | Default | Description |
|------|------|----------|---------|-------------|
| `tone` | `"default" \| "elevated" \| "data" \| "warning" \| "danger"` | no | `"default"` | Visual tone — token mapping in §6 |
| `padding` | `"sm" \| "md" \| "lg"` | no | `"md"` | Padding density — token mapping in §6 |
| `header` | `ReactNode` | no | — | Custom header slot; takes precedence over `headerTitle` (see §3.1) |
| `headerTitle` | `string` | no | — | Renders an uppercase title row (with a `▸` glyph, `aria-hidden`) when no `header` slot is given |
| `footer` | `ReactNode` | no | — | Optional footer slot, rendered below a top border (`mt-3 border-t border-border pt-3`) |
| `onClick` | `() => void` | no | — | When provided, the card becomes interactive — see §4 and §5 |
| `className` | `string` | no | — | Merged via `cn()` onto the root `<div>` |
| `children` | `ReactNode` | no | — | Card body content, rendered inside a plain `<div>` |
| *(rest)* | `ComponentProps<"div">` (minus the overridden `onClick`) | no | — | + native `<div>` attributes (passthrough via `{...props}`) |

## 3.1 Data Contract

**Cross-prop join rules:**

| Prop A | Field A | Prop B | Field B | Relationship |
|--------|---------|--------|---------|--------------|
| `header` | (`ReactNode` value) | `headerTitle` | (`string` value) | `header` takes precedence — when both are provided, `headerTitle` is ignored; the uppercase title row is rendered from `headerTitle` only as a fallback when `header` is `undefined` |

## 4. Component States

> No `useState`/`useReducer` — the interactive/static distinction is derived once per render from the presence of `onClick` (`typeof onClick === "function"`), and drives real visual + interactivity differences, so it is documented here.

| State | Trigger | Visual change | Interactivity |
|-------|---------|---------------|----------------|
| `static` | `onClick` is `undefined` (default) | Plain `<div>` with only `cardVariants({ tone, padding })` classes; no `role`, no `tabIndex`, no hover/focus styling beyond base tone | none |
| `interactive` | `typeof onClick === "function"` | Adds `role="button"`, `tabIndex={0}`, `cursor-pointer hover:bg-hover`, `focus-visible:outline-none focus-visible:ring-2 focus-visible:ring-ring`, and an `Enter`/`Space` keydown handler that invokes `onClick` | full |

| Event | Payload type | When emitted | Consumer action |

| Variant | Prop | Token classes | Usage context |
|---------|------|----------------|----------------|

## 7. Do / Don't

| Do | Don't |
|----|-------|
| Use `headerTitle` for a simple uppercase label row | Don't expect `CardHeader`/`CardTitle`/`CardContent`/`CardFooter` subcomponents — this is a single-component slot-prop API, not shadcn's compound pattern |
| Pass `onClick` only when the whole card should be one clickable/keyboard-activatable target | Don't pass `onClick` on a card that also contains its own nested interactive children (buttons/links) — the source has no guard against overlapping interactive targets |

## 8. BDD Scenarios

### Default render

```
Given a Card with default tone ("default") and default padding ("md") and children
When it mounts
Then it renders a static <div> with border-border bg-surface p-4, no role
  and no tabIndex
```

### Key state — interactive

```
Given a Card with onClick provided
When it renders
Then it exposes role="button" and tabIndex={0}, and clicking it calls
  onClick exactly once
```

### Keyboard navigation

```
Given an interactive Card (onClick provided) is focused
When the user presses Enter or Space
Then onClick fires via the component's own onKeyDown handler
  (event.key === "Enter" || event.key === " ")
```

## 9. Accessibility Contract

| Requirement | Implementation |
|-------------|-----------------|
| Label | No built-in `aria-label` — accessible name derives from `header`/`headerTitle`/`children` content <!-- TO CONFIRM: no explicit labelling strategy documented in source for interactive cards with no visible title --> |
| Keyboard | `Enter` and `Space` activate `onClick` when interactive; no keyboard handling when static |
| Focus management | `focus-visible:ring-2 focus-visible:ring-ring` applied only when interactive; no programmatic focus movement on any interaction |
| ARIA states | `role="button"` set only when `onClick` is provided; no `aria-pressed`/`aria-disabled` exist |
