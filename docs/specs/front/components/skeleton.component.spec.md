## 1. Purpose and Responsibilities

Renders a placeholder block (`<div>`) that pulses to indicate loading/reserved content area. Applies a pulsing background (`animate-pulse bg-elevated`, disabled under `prefers-reduced-motion` via `motion-reduce:animate-none`) and marks itself `aria-hidden="true"`. Sizing/shape (width, height, rounding) is entirely delegated to the consumer via `className`.

**Out of scope for this component:** actual content loading/fetching logic, layout sizing (no default width/height beyond what the consumer passes), announcing loading state to assistive technology (the element is `aria-hidden`, so a separate live region/status is the consumer's responsibility).

---

## 2. When to Use / When Not to Use

| Use when | Do not use when |
|----------|-----------------|
| Reserving visual space for content that is still loading (text lines, avatars, cards) to reduce layout shift (CLS) | Announcing a loading state to screen-reader users → pair with a visually-hidden `role="status"`/`aria-live` element, since `Skeleton` itself is `aria-hidden` <!-- TO CONFIRM: no dedicated loading-announcement component exists in this UI Kit yet --> |

## 3. Props Contract

| Prop | Type | Required | Default | Description |
|------|------|----------|---------|-------------|
| `className` | `string` | no | `undefined` | Merged with the base pulsing classes via `cn()`; consumer supplies width/height/rounding here |

`SkeletonProps` is a direct type alias: `HTMLAttributes<HTMLDivElement>` (no additional component-specific fields).

## 4. Component States

> No internal state (no `useState`/`useReducer`). Single visual presentation, always rendered the same way regardless of props other than `className`.

| State | Trigger | Visual change | Interactivity |
|-------|---------|---------------|----------------|
| `idle` | initial render | `animate-pulse bg-elevated` applied; pulse animation disabled (`motion-reduce:animate-none`) when the user prefers reduced motion | none (not interactive; `aria-hidden="true"`) |

## 7. Do / Don't

| Do | Don't |
|----|-------|
| Set explicit width/height via `className` (e.g. `h-4 w-40`) to match the shape of the content being loaded | Rely on Skeleton for any default size — it renders as a zero-size `<div>` unless `className` supplies dimensions |
| Stack multiple `Skeleton` elements to mimic multi-line text or avatar+text layouts (see `TextLines` story) | Nest interactive content inside `Skeleton` — it is `aria-hidden="true"` and purely decorative |

## 8. BDD Scenarios

### Default render

```
Given the component receives a className with explicit height/width (e.g. "h-4 w-40")
When it mounts
Then it renders a <div> with animate-pulse bg-elevated classes, the consumer's size classes, and aria-hidden="true"
```

### Reduced motion

```
Given the user's OS/browser has prefers-reduced-motion enabled
When the Skeleton mounts
Then the motion-reduce:animate-none class suppresses the pulse animation while bg-elevated remains visible
```

### Keyboard navigation

```
Given the Skeleton is rendered on the page
When the user tabs through the page with the keyboard
Then focus never lands on the Skeleton — it has no focusable/interactive role and is aria-hidden="true"
```
