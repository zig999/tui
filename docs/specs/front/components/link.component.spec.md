## 1. Purpose and Responsibilities

Renders a native `<a>` anchor styled with the kit's semantic tokens (primary color, underline-on-hover, focus ring). Supports an `external` flag that, when `true`, adds `target="_blank"` and `rel="noopener noreferrer"` for security, and appends a visible `↗` glyph after the children to signal that the link leaves the current context.

**Out of scope for this component:** client-side routing integration (renders a plain `<a href>`, not a router `Link`/`NavLink` — no TanStack Router integration is wired in); download-link semantics (no `download` handling beyond native passthrough).

## 2. When to Use / When Not to Use

| Use when | Do not use when |
|----------|-----------------|
| Rendering an inline or standalone hyperlink to an internal route or an external URL | The action does not navigate anywhere (e.g. a clickable action that submits a form or triggers a handler) → use `Button` with `variant="ghost"` instead |

## 3. Props Contract

```ts
import type { AnchorHTMLAttributes } from "react";
```

| Prop | Type | Required | Default | Description |
|------|------|----------|---------|-------------|
| `href` | `string` | yes | — | Anchor destination. Declared required on `LinkProps` even though `AnchorHTMLAttributes` itself marks it optional. |
| `external` | `boolean` | no | `false` | When `true`, adds `target="_blank"` and `rel="noopener noreferrer"` to the rendered `<a>`, and renders an `aria-hidden="true"` `↗` glyph after `children`. |
| `children` | `ReactNode` | no | — | Link content, rendered before the optional external-indicator glyph. |
| `className` | `string` | no | — | Merged via `cn()` with the component's base classes. |
| ...rest | `AnchorHTMLAttributes<HTMLAnchorElement>` | no | — | + native `a` attributes (passthrough) — e.g. `target`, `rel`, `download`, `aria-*`. Note: if `external` is `true`, the internally computed `target`/`rel` are spread onto the anchor *before* `...rest`, so an explicit `target`/`rel` passed by the consumer will override the `external`-derived values (JSX prop spread order: `{...externalProps} {...rest}`). |

`LinkProps` does not declare a `ref` prop; it is not destructured or forwarded explicitly in the component body (no `ref` in the destructuring list), so a `ref` passed by a consumer would only be forwarded if captured by `...rest` and passed to the underlying `<a>` — since `ref` is not a regular HTML attribute captured by object rest/spread in a function component signature, passing `ref` here has no special handling in this component (unlike `Button`, `RadioGroup`, etc., which explicitly destructure and forward `ref`). <!-- TO CONFIRM: whether Link should explicitly accept and forward `ref` per the Component Contract (all other UI-kit components in this batch destructure `ref` as a normal prop); as written, Link diverges from that pattern. -->

## 4. Component States

> No internal `useState`/`useReducer`, and no state-like attribute is computed by the component itself beyond the conditional glyph render — no internal state.

| State | Trigger | Visual change | Interactivity |
|-------|---------|---------------|----------------|
| `idle` | initial render | `text-primary no-underline hover:underline`, no internal state | full |

## 7. Do / Don't

| Do | Don't |
|----|-------|
| Set `external` when `href` points off-site, to get the security `rel` attributes and the visual `↗` indicator automatically | Manually set `target="_blank"` without `rel="noopener noreferrer"` — pass `external` instead so both are applied together |
| Use for navigation (internal route or external URL) | Use as a styled button/click-handler trigger with no `href` semantics — use `Button` instead |

## 8. BDD Scenarios

### Default render

```
Given a Link with href="#" and children="View details"
When it mounts
Then it renders an <a href="#"> with classes text-primary no-underline hover:underline and no external indicator
```

### External link state

```
Given a Link with href="https://example.com" and external={true}
When it mounts
Then the anchor has target="_blank" rel="noopener noreferrer" and an aria-hidden "↗" span is rendered after the children
```

### Keyboard navigation (focus-visible)

```
Given the Link is rendered in a page with other focusable elements before it
When the user presses Tab to reach it
Then the link receives focus (toHaveFocus) and shows the focus-visible ring via outline-none focus-visible:ring-2 focus-visible:ring-ring
```

## 9. Accessibility Contract

| Requirement | Implementation |
|-------------|----------------|
| Label | Accessible name comes from `children` (link text content); no separate `aria-label` is set internally |
| Keyboard | Native `<a href>` semantics — focusable and activatable via Enter, no custom key handling |
| Focus management | No custom focus trapping. Focus-visible ring via `outline-none focus-visible:ring-2 focus-visible:ring-ring` on the anchor itself |
| ARIA states | `aria-hidden="true"` on the decorative `↗` glyph span so it is not announced by assistive tech; no other ARIA state is set internally (native `rel="noopener noreferrer"` is a security attribute, not ARIA) |
