## 1. Purpose and Responsibilities

A styled tri-state checkbox built on a visually-hidden native `<input type="checkbox">` (`peer sr-only`) plus a decorative `<span data-slot="checkbox" aria-hidden="true">` box that carries all the visual styling, wrapped together in a single `<label>`. Supports a boolean `checked` value and an `"indeterminate"` state, the latter set imperatively on the native input's DOM property (not exposed as an HTML/ARIA attribute by this component).

**Out of scope for this component:** rendering its own text label styling beyond a plain inline `text-sm` â€” `children` is rendered as-is inside the `<label>`; grouping/"select all" indeterminate-derivation logic (the caller must compute and pass `"indeterminate"` itself).

---

## 2. When to Use / When Not to Use

| Use when | Do not use when |
|----------|-----------------|
| A single boolean or tri-state (indeterminate/"select all") toggle tied to a form field | The choice is binary and should read as an on/off control rather than a form checkbox â†’ use `switch` instead |
| The control must participate in native HTML form semantics (`<input type="checkbox">`) | Selecting exactly one value from a list of mutually-exclusive options â†’ use `radio-group` or `select` instead |

## 6. Variants and Compositions

> No `cva()` call exists in `checkbox.tsx` â€” there is no `variant`/`size` prop. Token classes are applied per prop-driven state on the decorative `<span data-slot="checkbox">`:

| Variant | Prop | Usage context |

| `disabled` | `disabled === true` | Label: `cursor-not-allowed opacity-50` |

## 7. Do / Don't

| Do | Don't |
|----|-------|
| Pass `checked` as `boolean \| "indeterminate"` and update it from `onChange` â€” the component is fully controlled | Don't expect the component to derive `"indeterminate"` from a list of child items itself; compute it in the consumer |
| Pass `children` for the visible label text so it sits inside the same clickable `<label>` | Don't rely on `aria-label` alone when `children` is omitted â€” there is no other accessible name source |

## 8. BDD Scenarios

### Default render

```
Given the component receives checked: false and a valid onChange
When it mounts
Then it renders an unchecked box (border-border) with the children text beside it
```

### Indeterminate / disabled state

```
Given checked: "indeterminate"
When it mounts
Then the box renders active styling (bg-primary) with a Minus icon and input.indeterminate is true
```

```
Given disabled: true
When it mounts
Then the label shows cursor-not-allowed and opacity-50, and the native input is disabled
```

### Keyboard navigation

```
Given the checkbox is focused (native <input>, no custom key handling in this component)
When the user presses Space
Then the native browser toggles the input and fires onChange with the new checked value
```

## 9. Accessibility Contract

| Requirement | Implementation |
|-------------|---------------|
| Label | Native `<label htmlFor={id}>` wraps the input and renders `children` as visible text â€” standard HTML label association |
| Keyboard | No custom key handling; relies entirely on native `<input type="checkbox">` semantics (Space toggles, Tab moves focus) |
| Focus management | Focus ring rendered on the decorative span via `peer-focus-visible:*` when the underlying (visually hidden) input receives `:focus-visible` |
| ARIA states | `aria-invalid` accepted via passthrough (`...props`) and styled via `peer-aria-invalid:border-destructive`; the checked/indeterminate visual state is not exposed as `aria-checked` â€” it relies on the native `<input type="checkbox">`'s own checked/indeterminate semantics exposed to assistive tech |
