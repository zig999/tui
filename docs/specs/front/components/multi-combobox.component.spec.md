## 1. Purpose and Responsibilities

A multi-select combobox: an `input[role="combobox"]` that shows selected items as removable chips and opens a `ul[role="listbox"]` of candidate options filtered by the typed query. Selection is fully controlled (`value: string[]` / `onChange`); the component holds no selected-items state of its own, only the open/closed flag, the typed query, and the keyboard-active index. It always narrows `options` locally by `label` (client-side fallback filter) and optionally calls `onSearch(query)` on every keystroke so a caller can drive server-side search in parallel.

**Out of scope for this component:** fetching/paginating options (caller supplies the full `options` array via props — see MIGRATION note in source), the "creatable" inline create-new-item mode (dropped; required an async mutation, which belongs in a feature hook composed around this primitive, not in the primitive itself), debounce/`minChars` throttling of `onSearch` calls (caller's concern), and single-selection (see `select`).

---

## 2. When to Use / When Not to Use

| Use when | Do not use when |
|----------|-----------------|
| The user must choose zero or more values from a filterable, potentially large option list, shown as removable chips | Exactly one value must be selected → use `select` instead |
| Options fit in memory as a flat array (caller may still narrow the array server-side via `onSearch`) | Free-text tag creation (values not present in `options`) is required → not supported; the "creatable" mode was removed during migration <!-- TO CONFIRM: no replacement primitive currently exists for creatable multi-select --> |

## 3.1 Data Contract

**Field usage:**

| Field | Source prop | Internal use | Behavior if absent |
|-------|-------------|--------------|-------------------|
| `value` (option) | `MultiComboboxOption.value` (each item in `options`) | Identity key for `role="option"` items, compared against the top-level `value: string[]` to exclude already-selected options from the dropdown and to resolve chip labels | Required field — `MultiComboboxOption` has no optional `value` |
| `label` | `MultiComboboxOption.label` | Rendered as chip text and option row text; also the field the local client-side filter matches against (`option.label.toLowerCase().includes(queryTrimmed)`) | Required field |
| `sublabel` | `MultiComboboxOption.sublabel` | Rendered as a secondary muted/inverted line under the option label in the dropdown row only (not shown on chips) | Not rendered — option row shows only the label line |

**Cross-prop join rules:**

| Prop A | Field A | Prop B | Field B | Relationship |
|--------|---------|--------|---------|--------------|
| `value` | (each string in the array) | `options` | `MultiComboboxOption.value` | `matches on` — chips are rendered from `value.map(v => knownOptionsRef.get(v) ?? { value: v, label: v })`; a `value` entry with no matching current option (e.g. narrowed out by server-side search) falls back to a chip whose label is the raw id, sourced from an internal cache of previously-seen options (`knownOptionsRef`), not from the current `options` array |
| `value` | (array) | `options` | `MultiComboboxOption.value` | `filters by` — the dropdown (`availableOptions`) excludes any option whose `value` is already present in the top-level `value` array |

## 4. Component States

> Internal state: `open` (boolean), `query` (string), `activeIndex` (number, keyboard-highlighted option), plus a non-rendering ref cache (`knownOptionsRef`) used only for chip-label fallback (see §3.1).

| State | Trigger | Visual change | Interactivity |
|-------|---------|---------------|----------------|
| `idle` (closed) | initial render / dropdown closed | Field shows chips (if any) + input with placeholder (only when `value` is empty); `border-border` | full |
| `open` | input receives a non-empty value (`onChange` sets `open` when `val.length > 0`), `ArrowDown` while focused, or input regains focus while `query.length > 0` | `ul[role="listbox"]` mounts below the field (`absolute z-20 mt-1 max-h-60 ... border border-border bg-elevated`); input gets `aria-expanded="true"` | full |
| `active-index` | `ArrowDown`/`ArrowUp` while open, or mouse-enter on an option | The option at `activeIndex` gets `bg-primary text-primary-foreground`; `scrollIntoView({ block: "nearest" })` runs on change; `aria-activedescendant` on the input points to that option's id | full |
| `at-max-items` | `maxItems !== undefined && value.length >= maxItems` | Search input is unmounted (not just hidden); a `text-muted-foreground` hint "Limite de {maxItems} itens atingido" renders below the field; dropdown cannot open (`showDropdown` forced false) | none (for adding; chip removal still works) |
| `loading` | `loading === true` while dropdown is open | Dropdown shows a single row with a spinning `Loader2` (`animate-spin`, `motion-reduce:animate-none`) and "Buscando..." instead of the option list | none (options not interactive while shown) |
| `error` | `error` is a non-empty string | Field border switches to `border-destructive`; a `text-destructive` message renders below the field with `id={errorId}`, referenced by the input's `aria-describedby` | full (field remains interactive) |
| `disabled` | `disabled === true` | Root gets `pointer-events-none opacity-50`; input gets native `disabled`; chip remove buttons get `tabIndex={-1}` | none |
| `closed-on-blur` | `onBlur` fires and `relatedTarget` is outside the root container | `open` resets to `false`, `activeIndex` resets to `-1` | full |

> **Transition parameters:**

| Parameter | Formula / Value | Unit | Applies to state |
|-----------|----------------|------|-----------------|
| Dropdown max height | `max-h-60` (15rem) | rem | `open` |
| Dropdown offset | `mt-1` | rem | `open` |
| Chip remove icon size | `14` | px | all |
| Loading spinner size | `16` | px | `loading` |

## 5. Events Emitted

| Event | Payload type | When emitted | Consumer action |
|-------|-------------|--------------|-----------------|
| `onChange` | `(value: string[]) => void` | An option is selected (mousedown on a non-selected option, or `Enter` while `activeIndex` points to one) → appends its `value`; or a chip's remove button is clicked, or `Backspace` is pressed with an empty query and `value.length > 0` (removes the last item) | Update the controlled `value` prop to the emitted array |
| `onSearch` | `(query: string) => void` | Every keystroke in the input (`onChange` of the native input), and once with `""` immediately after a successful selection (query is reset) | Optionally refetch/narrow `options` server-side for the new query |

> No `cva()` call exists in `multi-combobox.tsx` — there is no `variant`/`size` prop. Visual differentiation is entirely state-driven (§4) or prop-driven (`error`, `disabled`). Token classes per state, for reference:

|---------|------|----------------|

| `option-active` | `activeIndex === index` | `bg-primary text-primary-foreground` |
| `option-default` | *(not active)* | `text-foreground hover:bg-hover` |

## 7. Do / Don't

| Do | Don't |
|----|-------|
| Provide a stable `value: string[]` and update it from `onChange` — the component is fully controlled and holds no selection state | Rely on the component to fetch or paginate `options` — it is purely presentational; fetch in a `features/*/api/` hook and pass the result down |
| Use `maxItems` to cap selection count when the domain requires it | Expect chip labels to always reflect the current `options` array — a value narrowed out by server-side search falls back to a cached label or the raw id (see §3.1) |
| Pair with `onSearch` when the option set must be re-fetched per keystroke | Assume `onSearch` disables the built-in local label filter — both run; `onSearch` is additive |

## 8. BDD Scenarios

### Default render

```
Given the component receives valid required props (value: [], options, onChange)
When it mounts
Then it renders an empty field with the placeholder and no chips, dropdown closed
```

### Error state

```
Given error is a non-empty string
When it mounts
Then the field border switches to border-destructive and a text-destructive message renders below the field, linked via aria-describedby; the field remains interactive
```

### Max items reached

```
Given maxItems is set and value.length >= maxItems
When it renders
Then the search input is unmounted, a "Limite de {maxItems} itens atingido" hint shows, and the dropdown cannot open
```

### Keyboard navigation

```
Given the input is focused and the dropdown is open with available options
When the user presses ArrowDown/ArrowUp
Then activeIndex moves (wrapping at the ends) and the active option is highlighted and scrolled into view
When the user then presses Enter
Then onChange is called with the active option's value appended, the query resets, onSearch("") fires, and the dropdown closes
When the user presses Backspace with an empty query and at least one selected value
Then the last selected value is removed via onChange
```

## 9. Accessibility Contract

| Requirement | Implementation |
|-------------|---------------|
| Label | Optional visible `<label htmlFor>` when `label` prop is set; component provides no fallback `aria-label` when `label` is omitted |
| Keyboard | `ArrowDown`/`ArrowUp` open the dropdown and move `activeIndex`; `Enter` selects the active option; `Escape` and `Tab` close the dropdown; `Backspace` on an empty query removes the last chip |
| Focus management | DOM focus stays on the input at all times; options are `<li>` elements, not natively focusable — the "active" option is `aria-activedescendant` + visual highlight, not a DOM focus move; clicking the field container focuses the input (unless disabled or at max items) |
| ARIA states | Input: `role="combobox"`, `aria-expanded`, `aria-controls={listboxId}`, `aria-activedescendant`, `aria-busy={loading}`, `aria-describedby` (when `error`); list: `role="listbox"`, `aria-multiselectable="true"`; each item `role="option"` with `aria-selected` (reflects keyboard-active, not selected-membership — selected items are excluded from the list entirely) |
