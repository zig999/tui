## 1. Purpose and Responsibilities

Hand-rolled (no Radix primitive) trigger + popover calendar. Renders a button showing the formatted selected date (or a placeholder); clicking it opens an absolutely-positioned popover containing a month grid, optional time steppers, and quick-pick shortcuts. Uses native `Date` arithmetic exclusively — no `date-fns` or similar library. Animates the popover's mount/unmount with `motion/react` (`AnimatePresence` + `motion.div`).

Exports two parts:
- `DatePicker` — the full component described in §3–§9.
- `DateTimePicker` — a thin wrapper: `export function DateTimePicker({ ref, ...props }: DateTimePickerProps) { return <DatePicker ref={ref} {...props} showTime />; }`. It forwards every prop except `showTime` (which its own prop type omits) and hardcodes `showTime={true}` on the underlying `DatePicker`.

Internally, the file also defines `TimeSpinInput` (an hour/minute stepper: increment/decrement buttons + a clamped numeric input) — a private, unexported helper used only when `showTime` is `true`. It is not part of the public surface and has no story of its own.

**Structural notes (from code, not styling choices):**
- The whole component (trigger + popover) is wrapped in one `<div data-slot="date-picker" className="relative w-full">` — this is the element `ref` and native `<div>` passthrough props attach to, **not** the trigger `<button>` (see §3, §9).
- The popover is `position: absolute` (not portaled) with `z-20` — a lower stacking context than `Sheet` (`z-40`/`z-50`) or `Dialog` (`z-50`); it can be visually covered by either.
- The calendar grid is always exactly 42 cells (6 rows × 7 columns): leading/trailing days from the adjacent months (`inMonth: false`) pad the current month's days (`inMonth: true`) to a fixed 6-week grid (`buildCalendarCells`).
- Day-of-week headers are single letters: `["D", "S", "T", "Q", "Q", "S", "S"]` (Dom/Seg/Ter/Qua/Qui/Sex/Sáb) — note `Q` and `S` each repeat (Quarta/Quinta, Segunda/Sábado), which is visually ambiguous but is the exact array in the code.
- Selected-date display format: `dd / mmm / yyyy` (e.g. `10 / jul / 2026`), or `dd / mmm / yyyy  hh:mm` (two spaces before the time) when `showTime` — always prefixed with `⌖ ` when a value is set. The default placeholder mirrors this: `"⌖ dd / mmm / aaaa"` or `"⌖ dd / mmm / aaaa  hh:mm"`.
- There is no focus trap and no `Tab` key handling inside the popover (contrast with `Sheet`/`Dialog`, which trap focus via Radix `FocusScope`) — see §9.

**Out of scope for this component:**
- Range selection (start/end date) — only a single `Date | null` value.
- Min/max date bounds or disabled-date predicates — every day in the 42-cell grid is always selectable.
- Locale switching — month names, day headers, and all labels are hardcoded pt-BR strings (project is single-owner, pt-BR only, i18n off).
- Portal rendering — the popover is a normal absolutely-positioned child, not rendered via a portal (contrast with `Sheet`/`Dialog`).

## 2. When to Use / When Not to Use

| Use when | Do not use when |
|----------|-----------------|
| Capturing a single date (optionally with time) via a compact trigger + popover calendar, in a form or filter | A date range (start/end) is needed → this component has no range mode, use a different component |
| The picker can tolerate no min/max bounds and no per-day disabling | Certain dates must be blocked from selection (e.g. past dates, holidays) → not supported by this component as written |

## 3. Props Contract

### `DatePicker`

| Prop | Type | Required | Default | Description |
|------|------|----------|---------|-------------|
| `value` | `Date \| null` | yes | — | Selected date, or `null` when empty. Controlled — the caller owns state; the component derives its internal view/focus/time state from it (see §4) |
| `onChange` | `(date: Date \| null) => void` | yes | — | Fired with the newly selected date, or `null` when cleared (see §5 for every trigger) |
| `placeholder` | `string` | no | `"⌖ dd / mmm / aaaa"` (or `"⌖ dd / mmm / aaaa  hh:mm"` when `showTime`) | Text shown in the trigger when `value` is `null` |
| `disabled` | `boolean` | no | `false` | Disables the trigger (`disabled` attribute + `disabled:pointer-events-none disabled:cursor-not-allowed disabled:opacity-50`); `handleOpen` returns early, so the popover can never open while `disabled` |
| `showTime` | `boolean` | no | `false` | Adds an hour/minute stepper (two `TimeSpinInput`s) below the calendar grid; also switches the display/placeholder format to include `hh:mm` |
| ...props | `Omit<ComponentProps<"div">, "onChange">` | no | — | + native `<div>` attributes, applied to the **root wrapper div** (not the trigger `<button>`) — `onChange` is omitted from the native passthrough because it is redefined above with the `(date: Date \| null) => void` signature instead of `DOMAttributes`'s generic `ChangeEventHandler`. `ref` is accepted as a normal prop (React 19), attached to the root div and internally merged (via a local `mergeRefs`) with a click-outside-detection ref |

### `DateTimePicker`

`DateTimePickerProps = Omit<DatePickerProps, "showTime">` — identical props to `DatePicker` above, minus `showTime` (the type forbids passing it; the wrapper always renders `<DatePicker {...props} showTime />`).

| Prop | Type | Required | Default | Description |
|------|------|----------|---------|-------------|
| `value` | `Date \| null` | yes | — | Same as `DatePicker.value` |
| `onChange` | `(date: Date \| null) => void` | yes | — | Same as `DatePicker.onChange` |
| `placeholder` | `string` | no | `"⌖ dd / mmm / aaaa  hh:mm"` | Same as `DatePicker.placeholder`, but always the time-inclusive default since `showTime` is forced |
| `disabled` | `boolean` | no | `false` | Same as `DatePicker.disabled` |
| ...props | `Omit<ComponentProps<"div">, "onChange">` | no | — | Same as `DatePicker` |

## 4. Component States

> All of the following are real internal `useState` in `DatePicker` (not Radix, not props) — this component owns significantly more local state than `Sheet`/`Dialog`.

| State | Trigger | Visual change | Interactivity |
|-------|---------|---------------|----------------|
| `idle` (closed) | `isOpen` is `false` (initial render, or after any close path) | Only the trigger button is visible, showing `value` (formatted) or `placeholder` | Trigger is focusable/clickable (unless `disabled`) |
| `open` | Trigger clicked while not `disabled` (`handleOpen` toggles `isOpen`) | Popover mounts (animated in via `motion/react`, see transition parameters below); `viewYear`/`viewMonth`/`focusedDate` are reset to `value ?? today` | Full — popover keyboard nav active (see §9), trigger still togglable |
| `viewYear` / `viewMonth` (internal) | `prevMonth`/`nextMonth` clicks, arrow-key navigation crossing a month boundary (`navigateFocusedDate`), popover re-opening, or `value` changing externally (synced via `useEffect([value, showTime])`) | Calendar grid recomputes for the new month/year (`buildCalendarCells`) | — |
| `focusedDate` (internal) | Popover opens (reset to `value ?? today`), or `ArrowLeft/Right/Up/Down` inside the popover | The matching day cell gets a `ring-2 ring-ring` highlight (visual only — see §9 for the real-DOM-focus caveat) | — |
| `timeHour` / `timeMinute` (internal, only meaningful when `showTime`) | Initialized from `value`'s hours/minutes (or `0`) on mount; re-synced from `value` via the same `useEffect` when `value` changes externally and `showTime` is true; changed via `TimeSpinInput` | `TimeSpinInput` numeric displays update; if `value` is already set, changing either one immediately fires `onChange` with a new `Date` combining `value`'s year/month/day with the updated hour/minute | Full (steppers always interactive when `showTime`) |

> **Transition parameters** (popover mount/unmount animation, `motion/react`):

| Parameter | Formula / Value | Unit | Applies to state |
|-----------|----------------|------|-----------------|
| `opacity` | `0 → 1` (enter), `1 → 0` (exit) | — | `open` (enter/exit) |
| `y` (vertical offset) | `-4 → 0` (enter), `0 → -4` (exit) | `px` | `open` (enter/exit) |
| `duration` | `0.15` | `s` | `open` (enter/exit) |
| `ease` | `"easeOut"` | — | `open` (enter/exit) |

## 5. Events Emitted

| Event | Payload type | When emitted | Consumer action |
|-------|-------------|--------------|------------------|
| `onChange` | `(date: Date \| null) => void` | (1) A day cell is clicked, or `Enter` is pressed while a cell is focused — `selectDate` fires with `startOfDay(date)` (time zeroed) or, if `showTime`, with `date`'s y/m/d + current `timeHour`/`timeMinute`. (2) A quick-pick button (`hoje`/`amanhã`/`+7d`) — same `selectDate` path with a computed date. (3) `limpar` (clear) — fires with `null`. (4) When `showTime` and `value` is already set, changing either `TimeSpinInput` — fires with a new `Date` combining `value`'s y/m/d with the updated hour/minute (does **not** fire if `value` is `null` — only local `timeHour`/`timeMinute` state updates until a day is selected) | Consumer updates the controlled `value` it owns |

## 6. Variants and Compositions

No `cva()` block exists — no `variant`/`size` prop. The only axis is the boolean `showTime`, plus the `DateTimePicker` preset that forces it.

| Variant | Prop | Usage context |
|---------|------|----------------|
| Date only | `showTime={false}` (default) | Selecting a calendar date with no time component; format `dd / mmm / yyyy` |
| Date + time | `showTime={true}` on `DatePicker`, or use `DateTimePicker` directly | Selecting a date and a specific hour/minute; format `dd / mmm / yyyy  hh:mm`, adds the `TimeSpinInput` pair below the grid |

## 7. Do / Don't

| Do | Don't |
|----|-------|
| Always pass a controlled `value: Date \| null` and update it from `onChange` | Mutate the `Date` passed as `value` in place — the component always constructs new `Date` instances |
| Use `DateTimePicker` when both date and time are required | Try to pass `showTime` to `DateTimePicker` — the type omits it; it is always `true` |
| Rely on `Enter`/Arrow keys for keyboard selection inside the open popover | Expect `Tab` to move between day cells — cells have `tabIndex={-1}`; there is also no `Tab` handling to keep focus trapped inside the popover (see §9) |
| Pass `id`/`aria-*` intending to label the **trigger button** | Assume native passthrough props land on the `<button>` — they land on the outer root `<div>` (see §3, §9) |

## 8. BDD Scenarios

### Default render

```
Given a DatePicker with value=null and no placeholder override
When it mounts
Then the trigger button shows "⌖ dd / mmm / aaaa" in muted-foreground color, and the popover is not in the DOM
```

### Disabled state

```
Given a DatePicker with disabled=true
When the user clicks the trigger button
Then the button has the disabled attribute and disabled styling, handleOpen returns early, and the popover never opens
```

### Keyboard navigation

```
Given the popover is open and focusedDate is the currently highlighted cell
When the user presses ArrowRight, then ArrowDown, then Enter
Then focusedDate moves one day forward, then seven days forward (advancing viewMonth/viewYear if the week crosses a month boundary), and Enter calls onChange with that date and closes the popover
```

### Escape closes without committing

```
Given the popover is open with a focusedDate that differs from the current value
When the user presses Escape
Then the popover closes (isOpen becomes false) and onChange is NOT called — the focused-but-uncommitted date is discarded
```

### Live time adjustment on an already-selected value

```
Given a DatePicker with showTime=true and a non-null value
When the user increments the hour via TimeSpinInput
Then onChange fires immediately with a new Date combining value's year/month/day with the updated hour, without requiring the user to reselect a day
```

## 9. Accessibility Contract

| Requirement | Implementation |
|-------------|-----------------|
| Label | The trigger `<button>`'s accessible name comes only from its visible text content (formatted value or placeholder) — there is no `aria-label` prop reachable on the button itself, since native passthrough props attach to the root `<div>`, not the button. <!-- TO CONFIRM: whether an explicit label/aria-label prop on the trigger button is expected — not present in current code. --> |
| Keyboard | Trigger: standard `<button>` keyboard activation (`Enter`/`Space`). Popover (once open, `tabIndex={0}` on the popover `<div>` itself): `ArrowLeft`/`ArrowRight` move `focusedDate` ±1 day, `ArrowUp`/`ArrowDown` ±7 days, `Enter` commits `focusedDate` via `selectDate`, `Escape` closes without committing. Day cells themselves have `tabIndex={-1}` — real `Tab` focus never lands on an individual cell; keyboard navigation is a virtual/roving highlight (`focusedDate` state + `ring-2 ring-ring`), not real per-cell DOM focus |
| Focus management | On open: a `useEffect` calls `popoverRef.current?.focus()` via `setTimeout(0)`, moving real DOM focus to the popover container (`role="dialog"`, `tabIndex={0}`). On close (`Escape`, click-outside, or any selection): **no code returns focus to the trigger button** — unlike `Sheet`/`Dialog`, there is no Radix-provided (or custom) focus-restoration path. <!-- TO CONFIRM: verify whether missing focus-restoration-to-trigger on close is intentional or a gap. --> There is also no focus trap: no `Tab` key handler exists, so `Tab` while the popover is open follows normal DOM order and can leave the popover while it remains visually open |
| ARIA states | Trigger: `aria-haspopup="dialog"`, `aria-expanded={isOpen}`. Popover: `role="dialog"`, `aria-label="Selecionar data"` — no `aria-modal`, since there is no real modality/focus trap. Day cells carry no `role="gridcell"`/`aria-selected`/`aria-current`; the grid itself has no `role="grid"` — it is a plain CSS grid of `<button>`s distinguished only by visual styling (selected: `bg-primary text-primary-foreground`; today: `border-primary`; outside-month: `opacity-40`) |
