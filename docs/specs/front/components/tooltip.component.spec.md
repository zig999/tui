## 1. Purpose and Responsibilities

Two-part component wrapping `@radix-ui/react-tooltip` (`^1.2.12`), with a **deliberately narrow** prop surface — neither part forwards native/Radix passthrough props beyond what is explicitly typed.

- `TooltipProvider` — thin wrapper around `TooltipPrimitive.Provider`. Forwards only `delayDuration` (default `200`, **overriding** Radix's own default of `700`). Does not forward `skipDelayDuration` or `disableHoverableContent` — both stay at Radix's own defaults (`300` and `false`) and are not configurable from this wrapper.
- `Tooltip` — composes `TooltipPrimitive.Root` + `TooltipPrimitive.Trigger` (`asChild`, wrapping `children`) + `TooltipPrimitive.Portal` + `TooltipPrimitive.Content` (`side`, `sideOffset={8}` hardcoded, styled `border border-border bg-elevated px-2 py-1 text-xs text-foreground`). Body text comes from the `content` prop (`string` only, not `ReactNode`). When `disabled` is `true`, `Tooltip` renders `children` directly with **no** Radix wrapper at all (`if (disabled) return <>{children}</>`) — no tooltip logic runs, not even mounted-but-hidden.

**Out of scope for this component:**
- Controlled open state — `Tooltip`'s own prop type has no `open`/`defaultOpen`/`onOpenChange`; the underlying `TooltipPrimitive.Root` is always used uncontrolled.
- Any Radix `Content` prop beyond `side`/`sideOffset` — no `align`, `alignOffset`, `collisionPadding`, `onEscapeKeyDown`, `onPointerDownOutside`, `forceMount`, `aria-label`, etc. are exposed; the component's destructuring has **no rest spread**, so nothing beyond `children`, `content`, `side`, `disabled`, `className` reaches Radix.
- Rich tooltip content — `content` is typed `string`, not `ReactNode`.
- Per-tooltip `delayDuration`/`disableHoverableContent` override — only `TooltipProvider` exposes `delayDuration`, and only that one field.

## 2. When to Use / When Not to Use

| Use when | Do not use when |
|----------|-----------------|
| Adding a short, plain-text hint tied to a specific trigger element, shown on hover/focus | The content needs to block interaction or require an explicit dismissal → use `Dialog` instead (`dialog.component.spec.md`) |
| | The content is rich (more than plain text) or interactive → not supported — `content` is typed `string` and the `Content` element receives no children beyond it |

## 3. Props Contract

### `TooltipProvider`

| Prop | Type | Required | Default | Description |
|------|------|----------|---------|-------------|
| `children` | `ReactNode` | yes | — | Subtree that can render `Tooltip` instances sharing this provider's delay behavior |
| `delayDuration` | `number` | no | `200` | Milliseconds before a hovered/focused trigger's tooltip opens. Forwarded as the only prop to `TooltipPrimitive.Provider` |

Must wrap any subtree using `Tooltip` — `TooltipPrimitive.Root` throws `` `Tooltip` must be used within `TooltipProvider` `` if there is no ancestor provider (Radix context has no default value for this scope).

### `Tooltip`

| Prop | Type | Required | Default | Description |
|------|------|----------|---------|-------------|
| `children` | `ReactNode` | yes | — | Trigger element. Passed as `TooltipPrimitive.Trigger`'s single child via `asChild` — must accept a ref |
| `content` | `string` | yes | — | Tooltip body text, rendered as `TooltipPrimitive.Content`'s only child |
| `side` | `"top" \| "right" \| "bottom" \| "left"` | no | `"right"` | Forwarded to `TooltipPrimitive.Content`'s `side` — **overrides** Radix's own default of `"top"` |
| `disabled` | `boolean` | no | `false` | When `true`, short-circuits to `return <>{children}</>` — no `TooltipPrimitive.Root`/`Trigger`/`Portal`/`Content` is rendered at all |
| `className` | `string` | no | `undefined` | Merged via `cn()` onto `TooltipPrimitive.Content`'s base classes |

No `...props` / rest spread exists in the destructuring — this is the component's complete prop surface; no other Radix `Root`/`Trigger`/`Content` prop is reachable.

## 4. Component States

> No `useState`/`useReducer` in either part. Radix's `TooltipPrimitive.Root` owns the open/closed state internally (via `useControllableState`, uncontrolled since `Tooltip` never passes `open`/`onOpenChange`). Listed here because it is the primary state axis of the component.

| State | Trigger | Visual change | Interactivity |
|-------|---------|---------------|----------------|
| `idle` (closed) | Initial render; trigger not hovered/focused | `TooltipPrimitive.Content` unmounted | `children` fully interactive |
| `open` | Pointer hovers, or keyboard focuses, the trigger for `delayDuration` ms (from `TooltipProvider`, default `200`) | `TooltipPrimitive.Portal` mounts `TooltipPrimitive.Content` at the resolved `side`, offset `8`px from the trigger | `children` remains interactive; tooltip content itself is not interactive (`disableHoverableContent` stays at Radix default `false`, so moving the pointer onto the tooltip keeps it open, but no interactive elements are placed inside it) |
| `disabled` | `disabled === true` | No Radix tree renders at all — only `children`, unwrapped | `children` fully interactive; tooltip never opens under any trigger event |

## 7. Do / Don't

| Do | Don't |
|----|-------|
| Wrap the app (or the relevant subtree) once in a single `TooltipProvider` before rendering any `Tooltip` | Render `Tooltip` without an ancestor `TooltipProvider` — Radix throws `` `Tooltip` must be used within `TooltipProvider` `` |
| Pass a single ref-accepting element as `children` (Radix `Trigger asChild` requirement) | Pass rich/interactive content via `content` — it is typed `string` and rendered as plain text inside `Content` |
| Use `disabled` to conditionally suppress a tooltip based on state (e.g. no hint needed once a value is set) | Expect `disabled` to just visually hide the tooltip — it prevents the entire Radix tree (`Root`/`Trigger`/`Portal`/`Content`) from mounting |

## 8. BDD Scenarios

### Default render

```
Given a Tooltip with content="Tooltip content" wrapping a <button>Hover me</button>, inside a TooltipProvider
When it mounts
Then the button renders normally and no tooltip content is present in the document
```

### Disabled — no tooltip

```
Given a Tooltip with disabled={true}
When the user hovers or focuses the trigger
Then no TooltipPrimitive.Root/Trigger/Portal/Content is rendered — only children — and no tooltip ever appears
```

### Keyboard / hover interaction

```
Given the Tooltip is enabled and rendered inside a TooltipProvider with delayDuration={0}
When the user hovers the trigger (userEvent.hover)
Then a role="tooltip" element becomes visible showing the exact content text, positioned at the configured side with an 8px offset
```

## 9. Accessibility Contract

| Requirement | Implementation |
|-------------|-----------------|
| Label | The `content` string is the tooltip's accessible text; Radix renders both the visible `Content` and a visually-hidden duplicate for the accessible name (confirmed by the component's own test needing to scope its query to `[data-slot='tooltip-content']` to avoid a double match) |
| Keyboard | Radix opens the tooltip on trigger focus (not just hover) and closes on blur or `Esc`, with no custom keyboard handling added by this wrapper |
| Focus management | No focus trapping or redirection — the tooltip is non-modal; focus stays on the trigger while the tooltip is open |
| ARIA states | `role="tooltip"` set by `TooltipPrimitive.Content`; the trigger gets Radix's implicit `aria-describedby` linking to the content while open |
